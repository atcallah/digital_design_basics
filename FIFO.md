<h1 id="synchronizers">Synchronizers</h1>
<p>In some cases we cannot ensure that data is synchronized to the constraints of our designed data path. When we sample any analog sensor, for example, how can we ensure that the data being sampled abides to the setup and hold time constraints of our sequential circuit? We must implement some new techniques in order to handle such cases. The following examples are two common cases where we need to implement synchronization logic.</p>
<ol>
<li><strong>Asynchronous to Synchronous communication</strong></li>
</ol>
<p>Examples include a push button pressed by a human, photodetectors, temperature sensors, and pressure sensors. All of these processes are gated by physical processes and not a clock.</p>
<ol start="2">
<li><strong>Clock Domain Crossings</strong></li>
</ol>
<p>In large designs it is common to have multiple clock domains. For a SoC you might have different clocking frequencies for a CPU as compared to memory or other ASIC blocks.</p>
<h2 id="brute-force-synchronizers">Brute-Force Synchronizers</h2>
<p>This is one synchronization method utilized in systems. The fundamental principal behind a brute-force synchronizer is we double our flip-flop samplers to reduce the likelihood of introducing an illegal state into our system.<br>
<img src="https://images.slideplayer.com/17/5380093/slides/slide_11.jpg" alt="Brute-Force Synchronizer"><br>
In the diagram above, FF1 samples the asynchronous signal. This signal is subject to the uncertainty of the real world. Consequently, there is a clear possibility that signal “A” can switch during the forbidden switching zone. But, in this case we design the system such that the metastable state has time to settle to a 0 or 1 in the “AW” period. FF2 then samples the signal at the stable state.<br>

How well does this design work? In other words, what is the probability that “AS” is in an illegal state after a transition of “A”?</p>

<h3 id="gray-codes">Gray Codes</h3>
<p>The entire discussion above on brute-force synchronizers is only relevant for single bit values. If we have a series of brute-force synchronizers for a counter, for example, if the switch from 7 to 8 triggers an illegal state, all four transitioning bits have a probability of being incorrect.</p>
<p>Gray coding is a method to ensure that only 1 bit switches per transition. Consequently, the probability of an indeterminate state is greatly reduced. If you notice in our counter example, when the counter is gray coded we will only have the possibility of two output state compared to 16 with all four bits switching.</p>

</table><h2 id="fifo-synchronizers">FIFO Synchronizers</h2>
<p>The following block diagram describes the functionality of an FIFO synchronizer. All data is written to the FIFO according to the write clock and read according to the read clock. As a result, there is no timing issues with the read and write of data in.</p>
<p><img src="https://vlsiverify.com/wp-content/uploads/2022/12/asynchronous-fifo-1024x578.gif" alt="enter image description here"></p>
<p>The key caveat of this type of circuit is that we need to ensure that the FIFO is never written to when its full and read from when its empty. This is achieved by keeping write and read pointers for both clock domains. When the write and read pointers are equal the FIFO is either full or empty. Note that we must pass these pointers to the counterpart’s domain using brute-force synchronization!</p>
<h3 id="fifo-sizing-examples">FIFO Sizing Examples</h3>
